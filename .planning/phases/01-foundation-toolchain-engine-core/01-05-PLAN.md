**Step 1 — Write `vercel.json`** (per RESEARCH.md §"Code Examples → `vercel.json`"):

```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite",
  "rewrites": [
    {
      "source": "/((?!sprites/|assets/|favicon\\.ico).*)",
      "destination": "/index.html"
    }
  ],
  "github": {
    "silent": false
  }
}
```

The negative lookahead in `source` preserves `/sprites/*` (Pokémon pixel sprites under `public/sprites/` — DATA-04) and `/assets/*` (fingerprinted Vite chunks) and `/favicon.ico` (browser default request). Everything else falls through to `/index.html` so the SPA boots.

**Step 2 — Update `.gitignore`** to exclude Vercel's local link cache (the `.vercel/` directory created by `vercel link`):

```diff
node_modules
dist
.cache/
+ .vercel
```

**Step 3 — Build clean and verify the disclaimer is in the bundle:**

```bash
rm -rf dist
npm run build

# Sanity: the disclaimer must appear in the production bundle
grep -rq "non-commercial fan project" dist || (echo "ERROR: disclaimer missing from build output"; exit 1)

# index.html must exist
test -f dist/index.html || (echo "ERROR: dist/index.html not found"; exit 1)

echo "Build OK; disclaimer present."
```

If `grep` fails: re-check that plan 04's `Disclaimer.tsx` and `App.tsx` correctly import + render. Plan 04 already has a `tests/app.test.tsx` that asserts the text renders; if that's green, this should be green too.

**Step 4 — Install Vercel CLI (if not already on the system) and deploy:**

```bash
npx -y vercel --version    # confirm CLI is reachable
```

If the CLI prompts for login (it will, on first run), the executor MUST stop and create a `checkpoint:human-action` (already pre-declared in Task 2 below) — interactive auth is one of the rare cases where Claude cannot proceed alone (per checkpoints documentation: "Email verification, SMS 2FA, manual account approvals" — Vercel CLI login uses an OAuth flow with a one-time email link).

If the CLI is already authenticated (`vercel whoami` returns a username), proceed:

```bash
# First-time link: associates the cwd with a Vercel project.
# --yes accepts defaults: project name = directory name = "PokeTrail" (or "poketrail")
# --confirm is a synonym in older CLI versions.
npx vercel link --yes 2>&1 || true

# Deploy to production (--prod). For the very first run Vercel will create the project
# and provide a *.vercel.app subdomain (D-04 — default subdomain, no custom domain).
DEPLOY_OUTPUT=$(npx vercel --prod --yes 2>&1)
echo "$DEPLOY_OUTPUT"

# Extract the production URL (last *.vercel.app line)
DEPLOY_URL=$(echo "$DEPLOY_OUTPUT" | grep -oE 'https://[a-z0-9-]+\.vercel\.app' | tail -1)
echo "DEPLOY_URL=$DEPLOY_URL"

if [ -z "$DEPLOY_URL" ]; then
  echo "ERROR: could not parse deploy URL from output"
  exit 1
fi

# Wait briefly for CDN propagation, then verify the disclaimer is served
sleep 5
curl -sL "$DEPLOY_URL" | grep -q "non-commercial fan project" \
  && echo "OK — disclaimer served at $DEPLOY_URL" \
  || (echo "ERROR — disclaimer NOT served at $DEPLOY_URL (CDN delay or build issue)"; exit 1)

# Save the URL for Task 2 (README update)
echo "$DEPLOY_URL" > .vercel/.last-deploy-url 2>/dev/null || echo "$DEPLOY_URL"
```

**Important:** if the user has not yet pushed the repo to GitHub, the `vercel link` step above will create a Vercel project pointing at the local cwd, but auto-deploy-on-push won't be active until the user either (a) connects the project to GitHub via the Vercel dashboard or (b) installs the Vercel GitHub App. The CLI deploys we just ran are manual; auto-deploys come from the Git integration. We document this in Task 2's checkpoint.

If `npx vercel` errors with "not logged in" or similar interactive auth prompt, STOP — Task 2 is the human-action checkpoint that handles this.

<execution_notes>
- **Wave-0 probe-run pattern (Phase-1 `golden-baseline.ts` parallel):** The probe-run is a
  deliberately two-step process: first run the test with placeholders to capture the actual
  output, then commit the captured values as hard asserts. This is **the only way** to lock
  outputs of a deterministic-but-complex computation (the alternative — hand-computing expected
  HPs through 5–10 turns of Smogon-formula damage rolls — is infeasible and error-prone).
- **What "drift" means and why it's serious:** If a future plan changes the RNG-consumption
  order in any handler — even by adding a new `rng.next()` call somewhere subtle — the locked
  `rngCounter` value will not match. The test fails immediately, surfacing the regression at
  the exact PR that introduced it. This is the regression-prevention payoff of the Wave-0 lock.
- **What to do if the test fails after legitimate behavior change:** This is intentional design.
  When you genuinely want to change battle behavior (e.g., flip event order in Phase 4 because
  UI demands it), re-run the probe-run pattern: temporarily re-add the console.log + soft
  asserts, capture new values, replace hard asserts. The two-commit pattern keeps the audit
  trail clean (commit 1 = "capture new baseline", commit 2 = "lock new values").
- **Why three it-blocks instead of one:** The first locks specific values (most fragile). The
  second locks structural invariants (turnStart/turnEnd balance — won't break on minor RNG
  changes, only on emission-logic bugs). The third proves determinism orthogonally (runs the
  scenario twice, asserts equality — catches non-determinism bugs that ALSO match the locked
  values, which is rare but possible).
- **Why no separate `battleResolution.test.ts` or `ai.test.ts` updates here:** Those are owned
  by Plan 02-02. Plan 02-03 is intentionally narrow — only the integration scenario. If the
  resolution-order tests start failing during this plan's execution, that's a Plan 02-02 bug
  to fix retroactively, not a Plan 02-03 concern.
- **Don't over-engineer the histogram assertion:** A `toEqual({turnStart: N, turnEnd: N, ...})`
  for the captured histogram is fine. Don't try to predict counts analytically — let the
  probe-run produce them.
- **No source-code changes in this plan:** All source files (battleMachine.ts, ai.ts,
  battleResolution.ts, types.ts, battle-fixtures.ts) are stable from Plan 02-02. If you find
  yourself modifying them during Plan 02-03, STOP — you've found a bug in Plan 02-02 and should
  surface it as a "Deviations from Plan" entry in the eventual SUMMARY.md.
- **Probe-run hygiene:** When capturing JSON, save it as a comment block at the top of the
  describe IMMEDIATELY (before running Task 2's edits) — losing the captured JSON to a terminal
  scrollback after a session crash is the kind of small operational mistake that costs 10
  minutes per recovery. Treat it as a load-bearing artifact.
</execution_notes>
